# Crawler & Indexability Checklist

How the site owner can verify Googlebot and the AdSense crawler can actually reach and read sounez.com. Every item below includes the exact command or URL to check it with — run these against the **live production domain**, not localhost, since that's what Google actually crawls.

---

## 1. The production site returns HTTP 200

```bash
curl -sI https://www.sounez.com/
curl -sI http://sounez.com/          # should 301/308 → https://www.sounez.com/
curl -sI https://sounez.com/         # should 301 → https://www.sounez.com/
```

Expect `200 OK` on the `www` HTTPS URL, and a single-hop redirect (not a chain) from the apex/HTTP variants. This site's `next.config.ts` already enforces this (`redirects()` — apex→www and http→https, both in one hop, plus HSTS). Confirmed working on the live domain during this pass (checked earlier in this engagement: `http://sounez.com` → 308 → `https://sounez.com` → 301 → `https://www.sounez.com`, 200 OK, HSTS present).

## 2. Important pages are publicly accessible (no auth wall)

```bash
for path in / /tools /blog /smart-packs /about /contact /privacy-policy /terms-of-service /cookie-policy /dmca /faq; do
  echo "$path: $(curl -s -o /dev/null -w '%{http_code}' https://www.sounez.com$path)"
done
```

Every path above must return `200` with no login prompt, no Cloudflare/WAF challenge page, and no basic-auth prompt. None of these routes require a session in this codebase — confirm nothing was added at the hosting layer (Vercel deployment protection, a password-protect toggle, etc.) that would block anonymous requests, since that's a hosting-config setting outside this repo and this checklist can't verify it for you.

## 3. Googlebot specifically is not blocked

- Open `https://www.sounez.com/robots.txt` in a browser and confirm the `User-agent: *` block **allows** `/` and only disallows `/admin/` and `/api/` (see `app/robots.ts` / `lib/route-policy.ts` for the source of truth).
- In **Google Search Console** → URL Inspection, test `/`, `/tools`, one tool page, `/blog`, one article, and `/about`. Each should report "URL is on Google" or "Crawled — currently not indexed" (not "Blocked by robots.txt" or "Blocked due to unauthorized request").
- If the site sits behind a WAF/CDN (Cloudflare, etc.) with bot-fight-mode or "Under Attack Mode" enabled, that can block Googlebot even when `robots.txt` allows it — this is a hosting-dashboard setting, not something in this repo, so check it directly in that provider's dashboard and allowlist Googlebot's published IP ranges or user-agent if a bot-management rule is active.
- This site does **not** use any bot-blocking middleware (`middleware.ts` does not exist in this repo) and does not gate any public route behind authentication.

## 4. The sitemap is available and valid

```bash
curl -s https://www.sounez.com/sitemap.xml | head -20
curl -s https://www.sounez.com/sitemap.xml | grep -c '<loc>'
```

Expect a 200 response with valid XML and roughly ~76-80 `<loc>` entries (exact count moves as tools/posts are added — cross-check against `npm run quality:audit`'s reported route count, which read 77 known public routes at the time of this audit). Confirm `/admin` and `/api` do **not** appear anywhere in the output — `app/sitemap.ts` excludes them programmatically via `lib/route-policy.ts`, but re-verify on the live file since this is generated at request time.

Also submit the sitemap in Google Search Console (Sitemaps → enter `sitemap.xml` → Submit) if it hasn't already been submitted.

## 5. `robots.txt` is available

```bash
curl -sI https://www.sounez.com/robots.txt   # expect 200
curl -s  https://www.sounez.com/robots.txt    # expect the sitemap line at the bottom
```

Confirm the last line reads `Sitemap: https://www.sounez.com/sitemap.xml` with the correct production host (not `localhost` — this is generated from `NEXT_PUBLIC_SITE_URL`/`APP_URL`, so if it ever shows the wrong host, that env var is misconfigured in production; see §9 below).

## 6. Canonical URLs are correct

Spot-check page source (View Source, not DevTools-rendered) on a handful of pages:

```bash
curl -s https://www.sounez.com/tools/image-compressor | grep -o '<link rel="canonical"[^>]*>'
curl -s https://www.sounez.com/blog/how-to-compress-images | grep -o '<link rel="canonical"[^>]*>'
```

Every canonical must point to the `https://www.sounez.com/...` URL of that exact page (self-referencing), never to `localhost`, a preview deployment URL, or the apex (non-`www`) domain. This is generated by `lib/site-url.ts`'s `getSiteUrl()`, which reads `NEXT_PUBLIC_SITE_URL` (falling back to `APP_URL`, then `http://localhost:3000` only if both are unset) — a canonical pointing at `localhost` in production means that env var isn't set on the live deployment. This was verified correct on the live site during this engagement, but re-check after any redeploy or environment change.

## 7. No important page contains `noindex`

```bash
for path in / /tools /blog /about /contact /privacy-policy /terms-of-service /faq; do
  echo "$path:"; curl -s https://www.sounez.com$path | grep -io 'name="robots"[^>]*' || echo "  (no robots meta tag — indexable by default)"
done
```

None of the public pages above should print a `noindex` directive. This site's indexation policy is centralized in `lib/route-policy.ts` (`NOINDEX_PATHS = ["/admin", "/api"]`) — only `/admin` currently sets it via an explicit `<meta name="robots">` (`app/admin/layout.tsx`); `/api` relies on `robots.txt` disallow plus not being linked anywhere public. If you ever see `noindex` on a tool, blog, or trust page, check whether that route's parent layout was accidentally nested under `app/admin/` or copied its metadata.

## 8. The site works without authentication

Already covered in §2 — every route this repo defines is public by default. The only routes requiring a credential are `app/api/admin/comments/*`, which check a bearer token server-side (`lib/admin-auth.ts`) and are correctly excluded from the sitemap and marked `noindex`. If Googlebot or the AdSense crawler ever receives a 401/403 on a *public* page, the cause is a hosting-layer setting (see §3), not application code.

## 9. No redirect loops

```bash
curl -sIL https://www.sounez.com/ | grep -i '^HTTP'
curl -sIL https://sounez.com/tools | grep -i '^HTTP'
curl -sIL http://www.sounez.com/blog | grep -i '^HTTP'
```

`-L` follows redirects; `-I` shows only headers. Each command should show at most **one** redirect hop before a final `200`. The redirect rules live entirely in `next.config.ts` (`redirects()`): legacy root-level tool slugs → `/tools/{slug}`, the merged blog post → its replacement, `/terms` → `/terms-of-service`, and host/protocol normalization (apex → www, http → https). These are all single-hop by design — a loop would only appear if a hosting-level redirect (e.g. a CDN rule) was added on top of these and conflicts with them; that's outside this repo, so verify directly against the live domain after any hosting/DNS change.

## 10. Production environment variables (things that silently break the above if missing)

These are read at build/runtime and directly affect the checks above — confirm they're set in the actual hosting provider's environment, not just `.env.example`:

| Variable | Affects | Symptom if missing |
|---|---|---|
| `NEXT_PUBLIC_SITE_URL` (or `APP_URL`) | Canonical URLs, sitemap, robots.txt sitemap line, JSON-LD, OG URLs | Falls back to `http://localhost:3000` — every canonical/sitemap entry would point at localhost. **Highest-impact one to verify.** |
| `DATABASE_URL` | `lastModified` dates in the sitemap (optional — falls back to static dates if unset, does not break the sitemap) | No crawl-blocking effect either way; only affects `lastModified` freshness signals. |
| `NEXT_PUBLIC_ADSENSE_PUB_ID` / `NEXT_PUBLIC_ADSENSE_ENABLED` | Whether ads load and whether the AdSense site-ownership meta tag renders | Not indexation-blocking, but confirm intentional — see `ADSENSE_READINESS.md`. |

## 11. Server-side rendering isn't silently broken

```bash
curl -s https://www.sounez.com/tools/image-compressor | grep -c '<h1'
curl -s https://www.sounez.com/blog/how-to-compress-images | grep -c '<h1'
```

Both should return `1` (via `curl`, i.e. from the raw server-rendered HTML, not a JS-executed DOM) — if the count is `0`, the page is relying on client-side rendering for content Google needs, which this site does not do (all tool/blog/category pages are server-rendered or statically generated; only the interactive tool widgets themselves are client components, mounted below already-present server-rendered editorial content). A `0` here would indicate a regression, not expected behavior.

## 12. JavaScript-only content risk

This site's tool *pages* (editorial content, FAQs, examples) are server-rendered — confirmed via `ToolPageShell`/`ToolPageSections` being server components. Only the interactive tool widget itself (upload button, live preview, etc.) is a client component (`ToolClientRenderer.tsx`'s `next/dynamic` registry), which is expected and fine: Googlebot doesn't need to execute the compressor to index the surrounding article content. No page in this repo depends entirely on client-side rendering for its indexable text.

---

### Quick reference: all checks in one pass

```bash
BASE=https://www.sounez.com
echo "-- status codes --"
for p in / /tools /blog /smart-packs /about /contact /privacy-policy /terms-of-service /cookie-policy /dmca /faq /sitemap.xml /robots.txt /ads.txt; do
  echo "$p: $(curl -s -o /dev/null -w '%{http_code}' $BASE$p)"
done
echo "-- redirect hops --"
curl -sIL http://sounez.com/ | grep -ci '^HTTP'
echo "-- sitemap URL count --"
curl -s $BASE/sitemap.xml | grep -c '<loc>'
echo "-- robots.txt sitemap line --"
curl -s $BASE/robots.txt | grep -i sitemap
```

Run this after every production deploy, before requesting (or re-requesting) an AdSense review.
